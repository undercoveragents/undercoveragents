# Monitoring and Troubleshooting

Use this skill for questions about where to confirm what happened after a user sends a message, runs a mission, or changes configuration.

## Main monitoring surfaces

- The dashboard helps users understand the current workspace and jump into the major resource areas.
- Inspector surfaces help review conversations, messages, and tool activity when someone needs to understand what happened in a chat.
- Mission runs, test suites, and job monitoring surfaces help confirm whether background work ran successfully or stalled.

## How to troubleshoot

- Start with the surface closest to the problem: a chat page for conversation issues, a mission run page for workflow issues, or a job monitor for background processing issues.
- Confirm whether the problem is configuration, execution, or visibility before suggesting edits.
- Use history and persisted run details to explain what the system actually did instead of guessing from the current page state alone.

## Practical guidance

- If the user asks whether something ran, point them to the matching run or transcript surface first.
- If the user asks why a response looks wrong, inspect the workflow or tool activity before rewriting configuration.
- Troubleshooting should move from evidence to action, not from assumptions to action.
